Eend

Author: Martijn Nieuwenhuizen

Date: 16-02-2016

Eend, is the name of the compagny Marrije works for (translated to Englisch it means Duck. A tippicle Dutch bird in case you don’t know it). Eend is a compagny who specialises in UX ad UI. They don’t do the design and the only thing they diliver in HTML is a prototype. So why hire them?

Marrije made that clear in the workshop she gave us. In this workshop, we had a fieuw printed versions from the website of the HvA (Collage of Amsterdam). And our assignment was clear: cut this page into blocks that can stand on thair own and still have a meening. So we did. The result was a bunch of little blocks from the website. But then it got difficult, and thairfor interesting. Because how far do you go. Is cutting out the blog fine, or do you need to sepparate the header from the intro and the body? So what does a piece needs to have to still have meening?

Wel this made me curious. Because if I work on a website i alway’s make a wireframe of the page and name everything. But this goed way further. If you apply this technique at the beginning of your proces you create ‘basic’ elements. This in my eyes is the definition of Object Orientated CSS. I’ll explain this in an example: I worked on a project for …. . We began this project with a style-guide. In this guide we try to create every element on the page. So what does a Header looks like, how does a list behave and wich buttons do we have? And whith the latest it went a bit wrong. Because we work with SASS, we had a style page?? named buttons.scss in the snippits map. In this document we declared the style of every button we needed on the page, it may have looked something like this:

// Default button
button {
border: none;
background-color: #99ff00;
padding: .3em .7em;
border-radius: 5px;
color: white;
}
button:hover {
color: black;
}

// Spesific buttons
[type=“submit”] {
background-color: green;
}
.disabled {
background-color: red;
}

So here comes the problem. Three months after the launge (livegang) I checked a little detail in the CSS and discovered some new files:

  • errorbutton.scss
  • red—light-button.scss
  • blog-button.scss
  • big-button.scss
  • small-button.scss

I think you catch my drift.

So why is this so bad.

If this contignues you wil have A LOT of new files, so your well thought of argitecture is screwed. That means that after three years, you will have 15 different buttons, 8 different news blocks with a different background and for that, 300 lines of useless striped thrue css.

This is where Eend comes in the picuture. They will analize your site for visual problems like this. But in my oppinion, the Front-enders are responsible for this fact. So, don’t create new files for every single thing you need to add for a client. First take a ook at the current structure and work with that!
